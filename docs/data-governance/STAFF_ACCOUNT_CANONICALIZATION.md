# تنظيم وتوحيد حسابات الموظفين

## الهدف

توحيد كل موظف على هوية أساسية واحدة مع الحفاظ على كل البيانات التاريخية، ومنع ظهور الحسابات القديمة أو المكررة في التقارير والواجهات.

## القواعد المعتمدة

1. `staff.id` هو الهوية الأساسية للموظف.
2. `staff_accounts.id` هو حساب الدخول فقط، وليس بديلًا عن `staff.id`.
3. الحساب الأساسي يجب أن يكون:
   - مرتبطًا بـ `staff_id` صحيح.
   - `active = true`.
   - `can_login = true`.
   - اسمه ودوره وفرعه مطابقون لملف الموظف.
4. الحساب القديم لا يُحذف مباشرة؛ يتم:
   - نقل المراجع إلى `canonical_staff_id` عند الحاجة.
   - تعطيل الدخول.
   - إخفاؤه من الإدارة اليومية.
   - الإبقاء عليه للأثر التاريخي والمراجعة.
5. لا يتم الدمج بالاسم وحده. التطابق يعتمد على:
   - `staff_id` أولًا.
   - الفرع.
   - نوع الدور.
   - اسم المستخدم.
   - آخر دخول.
   - البيانات المرتبطة في الجداول الأخرى.

## الحسابات المؤكدة حاليًا

- الصيدلي: `dr.eslam` — د إسلام فاروق — فرع شكري.
- الدليفري: `eslam.delivery` — إسلام السبع — فرع شكري.

يجب منع أي Alias يجعل الحسابين هوية واحدة لأنهما موظفان مختلفان.

## مراحل التنفيذ

### المرحلة 1 — Audit فقط

شغّل:

```text
supabase/migrations/20260713_staff_account_canonicalization_audit.sql
```

ثم صدّر النتائج التالية:

```sql
select * from public.staff_account_cleanup_audit
order by normalized_name, branch, role, created_at;

select * from public.staff_account_duplicate_groups
order by accounts_count desc, normalized_name;

select * from public.list_staff_identity_reference_columns();
```

لا يتم تعديل أي حساب في هذه المرحلة.

### المرحلة 2 — اعتماد خريطة الدمج

يتم إدخال كل حساب قديم والحساب الأساسي المقابل داخل:

```text
public.staff_account_merge_plan
```

ولا يتحول السجل إلى `approved` إلا بعد مراجعة:

- الاسم والفرع والدور.
- `staff_id` الحالي والأساسي.
- آخر تسجيل دخول.
- عدد المراجع في الفواتير والتقييمات والحضور والمهام والنقاط.

### المرحلة 3 — نقل المراجع

يتم إنشاء Migration منفصلة لكل مجموعة جداول، مع:

- نسخة احتياطية داخل جدول Audit قبل التحديث.
- تحديث `staff_id` فقط عند وجود تطابق مؤكد.
- عدم تغيير أسماء تاريخية نصية إلا لو كانت واجهة العرض تعتمد عليها.
- تسجيل عدد الصفوف قبل وبعد.

### المرحلة 4 — تعطيل الحسابات القديمة

بعد التحقق:

```text
active = false
can_login = false
visible_in_admin = false
```

ويُمنع حذف الحسابات من قاعدة البيانات في أول دورة تشغيل بعد الدمج.

### المرحلة 5 — توحيد التطبيق

كل الصفحات يجب أن تستخدم:

- `staff_id` للتجميع.
- `staffIdentityResolver` لحل الأسماء القديمة.
- حسابًا واحدًا فقط عند العرض.
- Badge واضح للحسابات غير المرتبطة أو غير المعتمدة.

## بوابة الأمان قبل الدمج

- لا يوجد حسابان قادران على الدخول لنفس الموظف.
- لا يوجد حساب `disabled` وما زال `can_login = true`.
- لا يوجد حساب نشط بدون `staff_id` إلا باستثناء موثق.
- د إسلام فاروق وإسلام السبع منفصلان بالكامل.
- التقارير لا تتغير إجمالياتها بعد نقل المراجع.
- الحضور والنقاط والتقييمات والفواتير لا تفقد أي صف.
- `npm run typecheck` و`npm run build` ناجحان.
